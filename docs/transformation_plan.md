# SanSi 美股研究工具改造计划书

> 版本：v1.0 | 日期：2026-07-22 | 作者：Product Team

---

## 一、概述

### 1.1 背景

SanSi（三思）是一个 AI 驱动的美股研究工具，目标是输入股票代码即可生成券商研报级别的深度分析报告。项目衍生自 AI4Finance Foundation 的 FinRobot，已完成初步改造：移除了多智能体框架（AutoGen）、将 LLM 切换为 DeepSeek、增加了 yfinance 作为备用数据源、搭建了 FastAPI + Vue 3 的 Web 应用骨架。

### 1.2 当前阶段判断

**项目处于"原型验证阶段"（Prototype）**：核心流程能跑通（输入 ticker → 输出报告），但报告深度、数据覆盖、产品体验三个维度均未达到可交付水平。本计划书的目标是将其从原型推进到 **MVP（最小可行产品）**，再到 **可运营产品**。

### 1.3 改造目标

| 维度 | 当前状态 | 目标状态 |
|------|---------|---------|
| 报告质量 | 8 段简单文本 + 11 张图 | 10 模块完整研报，对标 GS/MS Initiating Coverage |
| 数据覆盖 | FMP + yfinance 双源 | 多源数据融合（SEC EDGAR、行业数据、机构持仓） |
| 产品体验 | Vue 3 CDN 原型 | 工程化前端，报告管理、对比、导出 |
| 智能化 | 简单 prompt 一次性生成 | 多 agent 结构化生成 + 交叉校验 |
| 可运维性 | 单机 SQLite | 数据库迁移、日志系统、监控指标 |

### 2.2 技术债务

1. **LLM 生成质量瓶颈**：`text_generator_agents.py` 的 system prompt 过于简单（如 "write a comprehensive company overview (300-400 words)"），缺乏对输出结构、数据引用、分析深度的强制约束。
2. **两套 agent 系统并存**：`text_generator_agents.py`（直接调用）和 `equity_agents/`（OpenAI Agents SDK）各写了一遍，前者在用，后者未启用。重复维护成本。
3. **预测模型过于简单**：`revenue-growth-2025/2026/2027` 三个硬编码参数，无行业基准校验，无三情景自动生成。
4. **前端技术栈落后**：Vue 3 CDN 方式无法做工程化管理，无组件复用，无状态管理，无路由。
5. **无数据缓存层**：每次生成报告都重新调用 FMP API，既慢又浪费额度。
6. **错误处理不完善**：LLM 调用失败时的 fallback 文本质量很低（见 `_get_fallback_text`），基本不可用。
7. **报告模板与 PRD 不一致**：PRD 和 `equity_research_template.md` 定义了 10 模块，但实际文本生成只覆盖了 8 段，缺少"行业分析""技术面分析""后续关注信号"的 LLM 生成。

## 三、PRD 与代码的差距矩阵

对比 `equity_research_template.md` 中定义的 10 个模块与当前代码实现状态：

| # | 模块 | PRD 要求 | 代码实现 | 差距 |
|---|------|---------|---------|------|
| 1 | 投资概要 | 评级框 + 走势图 + 核心论点 + 财务快照 | 部分覆盖（tagline + investment_overview） | 缺少评级逻辑、目标价自动推导、财务快照表 |
| 2 | 投资逻辑与风险 | 多支柱逻辑 + 风险对称分析 | 部分覆盖（risks 简单列举） | 缺少投资逻辑（支柱式展开），风险分析无量化 |
| 3 | 行业分析 | 规模/格局/五力/趋势 | **基本缺失** | 无行业数据源，LLM 无法凭空生成 |
| 4 | 公司分析 | 业务/管理层/拆分/护城河/成长驱动 | 简单覆盖（company_overview） | 无管理层数据、无业务拆分数据、无护城河分析框架 |
| 5 | 财务分析 | 收入/盈利/杜邦/现金流/偿债/Opex | 部分覆盖 | 杜邦有、现金流有，但偿债和运营效率分析不完整 |
| 6 | 盈利预测 | 核心假设 + 预测损益表 + 三情景 | 部分覆盖 | 假设简单，三情景需手动开启 |
| 7 | 估值分析 | DCF + 可比 + 历史区间 + Football Field | 部分覆盖 | 缺少历史分位、Football Field 图、评级推导 |
| 8 | 技术面分析 | 趋势/关键价位/动量 | **缺失** | 图表有股价+均线+RSI，但无 LLM 文字分析 |
| 9 | 后续关注信号 | 关键跟踪变量与判断标准 | **缺失** | 无此模块 |
| 10 | 风险提示 | 四维风险（宏观/行业/经营/估值） | 简单覆盖 | 仅有 5 条简单风险列举，无四维分类，无概率量化 |

**结论：10 个模块中，3 个基本缺失，5 个部分覆盖但深度不足，仅 2 个（财务分析 + 图表）接近可用水平。**

#### 1.2 LLM 生成重构

**目标**：从"一次性简单 prompt"升级为"结构化、数据驱动、多轮优化的专业生成"。

**当前问题深挖**：

当前 `text_generator_agents.py` 的生成逻辑：

```python
# 当前方式（简化示意）
system_prompt = "You are a financial analyst. Write a company overview (300-400 words)..."
user_prompt = f"Company: {name}\n\nFinancial Metrics:\n{df.to_markdown()}"
response = openai.chat.completions.create(model=model, messages=[...])
```

问题：
1. 所有数据一视同仁地 dump 进 prompt，LLM 不知道哪些是关键信息
2. 输出格式无约束，可能遗漏必需章节
3. 没有要求引用具体数字（LLM 倾向于用模糊表述）
4. 没有交叉验证机制（估值数据和财务数据是否自洽？）

**改造方案**：

| 任务 | 描述 | 优先级 | 估时 |
|------|------|--------|------|
| 1.2.1 | **统一 Agent 系统**：合并 `text_generator_agents.py` 和 `equity_agents/`，保留一套（推荐用 OpenAI Agents SDK，更结构化） | P0 | 2d |
| 1.2.2 | **模块化 Agent 设计**：每个模块一个 agent，接收该模块专属的结构化数据摘要（不是全量 dump） | P0 | 3d |
| 1.2.3 | **Prompt 模板升级**：每个 agent 的 prompt 包含：① 角色定义 ② 输出结构约束（章节标题、表格格式、必须包含的数据点） ③ 数据引用要求（"每个结论至少引用 1 个具体数字"） ④ 质量自检清单 | P0 | 2d |
| 1.2.4 | **交叉校验层**：LLM 生成后，用规则引擎校验关键数字是否与财务数据一致（如报告中引用的 PE 是否与数据表中的匹配） | P1 | 2d |
| 1.2.5 | **多轮优化**：关键模块（投资概要、估值分析）走两轮生成——第一轮生成草稿，第二轮作为"审阅者"优化 | P2 | 2d |

**具体 Agent 设计（10 个模块 → 10 个 Agent + 1 个总编 Agent）**：

```
data (structured per module)
    ↓
┌──────────────────────────────────────────┐
│ Agent 1: 投资概要 (Investment Thesis)       │ ← 所有模块数据摘要
│ Agent 2: 投资逻辑 (Investment Rationale)    │ ← 行业 + 财务 + 估值数据
│ Agent 3: 行业分析 (Industry Analysis)       │ ← 行业数据专用包
│ Agent 4: 公司分析 (Company Analysis)        │ ← 公司数据 + 管理层数据
│ Agent 5: 财务分析 (Financial Analysis)      │ ← 财务数据专用包
│ Agent 6: 盈利预测 (Earnings Forecast)       │ ← 预测模型输出
│ Agent 7: 估值分析 (Valuation Analysis)      │ ← 估值引擎输出
│ Agent 8: 技术面分析 (Technical Analysis)    │ ← 技术指标数据
│ Agent 9: 关注信号 (Key Monitor Signals)     │ ← 风险 + 催化剂数据
│ Agent 10: 风险提示 (Risk Disclosure)        │ ← 风险分析数据
│ Agent E: 总编 (Editor-in-Chief)             │ → 统一风格、消除矛盾、撰写 tagline
└──────────────────────────────────────────┘
    ↓
quality check (rule-based + consistency)
    ↓
final report
```

## 五、目标技术架构

```
┌─────────────────────────────────────────────────┐
│                   Frontend (Vite + Vue 3 + TS)    │

└──────────────────────┬──────────────────────────┘
                       │ REST API (JSON)
┌──────────────────────┴──────────────────────────┐

│  │          │  │          │  │               │   │
│  │ FMP API  │  │ 10 Module │  │ HTML Renderer │   │

                       │

│              Data & Infrastructure                 │
│   PostgreSQL │ Data Cache │ File Storage │ Logs   │
└─────────────────────────────────────────────────┘
```

### 关键架构变更

| 变更 | 原因 |
|------|------|
| 前端 CDN → Vite 工程化 | 支持组件化、TS 类型安全、可测试、可扩展 |
| subprocess 调用 → Python 函数调用 | 当前 subprocess 方案日志丢失、错误传递差、无法复用数据 |
| 一次生成 → 管道式生成 | 数据获取 → 分析 → agent 生成 → 质检 → 渲染，每步有明确输入输出 |
| SQLite → PostgreSQL（生产） | 并发支持、数据可靠性（SQLite 本地开发保留） |
| 无缓存 → 数据缓存层 | 减少 API 调用成本，加速报告生成 |
| 简单 prompt → 结构化 agent | 提升报告质量和一致性 |

## 八、风险管理

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| LLM 报告质量达不到要求 | 中 | 高 | Phase 1.5 建立评测基准，如不达标则切换模型（Claude/GPT） |
| FMP 数据不完整 | 高 | 中 | 多数据源互补（FMP + EDGAR + yfinance），优先保证核心数据可用 |
| EDGAR 解析复杂度超预期 | 中 | 中 | 先用 `python-edgar` 库，如不行则用第三方 EDGAR API（如 sec-api.com） |
| 前端开发资源不到位 | 中 | 低 | 前端延后不影响 Phase 1 报告质量提升，可以先 CLI 使用 |
| API 成本过高 | 低 | 中 | 数据缓存层控制 FMP 调用量，LLM 调用做 token 预算控制 |
